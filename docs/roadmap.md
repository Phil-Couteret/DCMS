## Phase 3 — Test safety net (before touching the data model further)

**Status: done (2026-07-17). All three suites written and passing.**

1. ✅ Added backend unit tests for the security-critical code from phases 0–2: `public.decorator.spec.ts` and `jwt-auth.guard.spec.ts` (the `@Public()`/global-guard bypass logic), `tenant.interceptor.spec.ts` (8 cases covering the JWT-authoritative tenant fix — header/JWT mismatch rejection, deactivated-tenant rejection, superadmin platform-level default and header-switch, pre-login and default-tenant fallback), `users.service.spec.ts` and `partners.service.spec.ts` (6 cases each confirming `password_hash`/`api_secret_hash` never appear in any returned object, across every service method), and `create-booking.dto.spec.ts` (a `class-validator`-direct test confirming valid payloads pass, invalid UUID/enum/negative values are rejected, and an unexpected extra field is rejected the same way `forbidNonWhitelisted` rejects it live). All 6 suites (32 tests) pass. Note: `UsersService`/`PartnersService`/`TenantsService` import `bcrypt` directly for password/secret hashing, which ships a native binary compiled for the machine it's installed on — the tests use `jest.mock('bcrypt', ...)` to stub it out, so they run without needing a matching native build.
2. ✅ Wrote and verified `backend/test/unauthenticated-routes.e2e-spec.ts` (4/4 passing) and `backend/test/tenant-isolation.e2e-spec.ts` (6/6 passing).
   - **`unauthenticated-routes.e2e-spec.ts`.** Boots the real app (with `PrismaService` stubbed out, since a rejection happens in the guard before any DB call) and uses Nest's own `DiscoveryService`/`MetadataScanner` to enumerate every registered route, then asserts every route that isn't genuinely credential-free gets a 401 with no `Authorization` header. This distinguishes `@Public()` (skips only the global admin guard) from "no auth at all" — `partner/*`, `partner/bookings`, `partner/customers`, and `partner-invoices/my-invoices` are `@Public()` but carry their own `@UseGuards(PartnerAuthGuard | JwtPartnerGuard)`, discovered generically via Nest's `GUARDS_METADATA`, so they're correctly expected to still 401. Only `AppController` (root/health) and the 3 login handlers are treated as truly open. Because it discovers routes at runtime instead of a hand-maintained list, a new controller or endpoint that forgets a guard is caught automatically. One real bug surfaced and fixed while getting this to pass: e2e tests built via `Test.createTestingModule` skip `main.ts`'s imperative `app.useGlobalPipes(...)` call (only the DI-registered global guard applies automatically), so the login endpoints were reaching their handlers with an unvalidated empty body and 401-ing from their own "invalid credentials" logic rather than 400-ing on missing fields — indistinguishable from a guard rejection. Fixed by replicating `main.ts`'s exact `ValidationPipe` config in the test's `app.init()`.
   - **`tenant-isolation.e2e-spec.ts`.** Seeds two real tenants + a customer each via Prisma, signs real JWTs via `AuthService`, and asserts tenant A's token can't list, read, update, or delete tenant B's customer (all correctly 404, since `CustomersService` filters by `tenant_id` before any write), plus that a header disagreeing with the JWT still gets the Phase 0 403, and that tenant B can still read its own data normally. Verified locally against a real Postgres (Homebrew, on your Mac — separate from both the Zbox's production DB and any Docker Compose setup) once `DATABASE_URL` had a real password and `npx prisma db push` synced the schema. All 6 tests pass.
3. Both suites pass. The intent of this gate — proving the phase 0–2 security fixes actually hold under real requests, not just in isolated unit tests — is met. Phase 4 (`tenant_id` everywhere) is unblocked.

## Phase 4 — Full multi-tenant data model

**Status: code complete (2026-07-21). Not yet applied to any real database — see "before you deploy" below.**

1. ✅ Added `tenant_id` to the 14 tables that didn't have it (`bookings`, `staff`, `equipment`, `boat_preps`, `boats`, `dive_sites`, `customer_stays`, `customer_bills`, `partner_invoices`, `pricing_configs`, `customer_consents`, `customer_certifications`, `data_subject_access_requests`, `bono_usage`) in `schema.prisma`, plus the matching `tenants` back-relations and `idx_*_tenant` indexes. Hand-authored migration `backend/prisma/migrations/20260721140235_phase4_tenant_id_everywhere/migration.sql` (the sandbox this was written in still can't run `prisma migrate dev`/reach a Prisma engine binary — see the Phase 2.3 note above — so this was written directly, following the same DDL shape Prisma itself generates) adds each column, **backfills it from the table's existing `location_id` or `customer_id` relation** (both already resolve to a tenant via `locations.tenant_id`/`customers.tenant_id`), then adds the FK (`ON DELETE NO ACTION`, matching the other 9 tenant_id relations per the Phase 2.3 drift fix) and index.
2. **Deferred — composite uniqueness left as-is.** Per target-architecture.md the ideal is `(tenant_id, username)`/`(tenant_id, email)` on `users`, `(tenant_id, contact_email)` on `partners`, `(tenant_id, key)` on `settings`. Investigated and found this isn't a safe drop-in: `users.service.ts findByUsername()`, `partner-auth.service.ts`, and `settings.service.ts findByKey()` all do `findUnique({ where: { username/contact_email/key } })`, relying on that field being globally unique — with per-tenant uniqueness, two tenants could share a username/email and that lookup becomes ambiguous (or stops compiling against the Prisma client, since a non-unique field can't be a `findUnique` selector). Doing this properly means reworking login/lookup to resolve tenant context (subdomain/header) *before* the query, with no fallback when that context is missing — a real change to login behavior, not a schema tweak. Asked you directly; you chose to keep these three fields globally unique as they are today rather than take on that rework. `bill_number`/`invoice_number`/`code` (bonos) stay globally unique too, unchanged.
3. ✅ **Every service's queries now filter by `TenantContextService`'s tenant** — and this turned up more than just the 14 new tables. Fixed the same gap (previously *zero* tenant filtering) in: `staff.service.ts`, `equipment.service.ts`, `boats.service.ts`, `dive-sites.service.ts`, `boat-preps.service.ts`, `customer-bills.service.ts`, `partner-invoices.service.ts`, `consents.service.ts`, `bookings.service.ts` (switched from an indirect `locations: { tenant_id }` join to the new direct column), and — on the 9 tables that already had `tenant_id` from earlier phases but whose services never used it — `audit.service.ts`, `breaches.service.ts` (data breach records), `government-bonos.service.ts`, `settings.service.ts`, and `statistics.service.ts` (the admin dashboard's revenue/booking counts, previously summed across every tenant). Also fixed a real IDOR in `dsar.service.ts`: `getDsar(dsarId)` looked up a DSAR request by ID alone, ignoring the `customerId` route param entirely, so any authenticated caller could read another customer's (or tenant's) DSAR request by guessing its UUID — now scoped to both tenant and the route's `customerId`. `customer_stays`/`customer_certifications`/`pricing_configs`/`bono_usage` have no dedicated service (read-only via already-tenant-scoped relations, or genuinely unimplemented/unreferenced anywhere in `src/`) — nothing to change there. Every touched module's `*.module.ts` now imports `TenantModule` to get `TenantContextService` injected.
4. ✅ The superadmin cross-tenant switch mechanism itself already existed from Phase 0 (`tenant.interceptor.ts`: a superadmin JWT has `tenantId: null`, and can switch into a tenant via `X-Tenant-ID`/`X-Tenant-Slug`/subdomain). What Phase 4 adds is the audit logging that was missing: every time that switch resolves, a fire-and-forget `audit_logs` entry is written (`action: 'tenant_switch'`, the target tenant, the superadmin's user id, method+path) — fire-and-forget so it never adds latency to the request, and a logging failure can never break the request itself. Covered by a new assertion in `tenant.interceptor.spec.ts` (still 8/8 passing).
5. **Deferred (optional per this plan).** Postgres RLS as defence-in-depth — not started. Application-level filtering (item 3) is the enforcement mechanism for now; revisit RLS once that's run against production for a while.

**Before you deploy this phase:** the new `tenant_id` columns start `NULL` on every existing row. Run the new migration (`cd backend && npx prisma migrate deploy && npm run prisma:generate`) *before* deploying the updated service code — if the code ships first, every tenant-scoped list/read on these 14 tables will suddenly return nothing for existing data, because `where: { tenant_id: <context tenant> }` doesn't match `NULL`. This is the same "backend needs prisma:generate" step as Phase 2.3, plus this migration needs to actually **run** (not just `resolve --applied` like the baseline was) since it changes real data. `npx tsc --noEmit` currently shows exactly 15 errors, all "`tenant_id` does not exist on type ..." on the 14 new tables plus the extended e2e test — confirmed by inspection to be solely the stale generated client, not real bugs; they clear once `prisma generate` is re-run. The `tenant-isolation.e2e-spec.ts` suite was extended with a `staff` case (previously totally unscoped) as a regression guard for this phase — run it locally the same way as Phase 3's.

## Sequencing notes

- Phases 0–1 can start immediately and in parallel (different files, no schema risk).
- Phase 2 depends on phase 0 being deployed (don't tighten validation on endpoints that are still unauthenticated in prod).
- Phase 3 must exist before phase 4 — adding `tenant_id` everywhere without an isolation test suite risks shipping the exact bug being fixed, just moved.
- Phase 5 has no hard dependency on 3–4 and can be picked up whenever capacity allows.
